# Gauge Review — Specification Iteration 1

## Summary

The spec is well-structured and covers the user's verbatim feature description faithfully, but it is grounded on a load-bearing factual error about the codebase: `steel specify` is **not** a registered CLI command, so FR-8 / AC-7 / AC-8 / AC-9 invent a surface that doesn't exist. There are also several smaller-but-real defects: an inaccurate description of the current precondition check, an FR-3 bypass that is internally contradicted by its own clarification note, an FR-10 git-commit requirement that conflicts with the `autoCommit` config, and an `AskUserQuestion` reference that is alien to the rest of `resources/commands/`. These need correction before clarification.

## Issues

### BLOCKING

- [B-1] FR-8 / AC-7 / AC-8 / AC-9 assume `steel specify <description>` is a registered CLI command. It is not. `src/cli.ts` lines 48–52 explicitly state: *"Per-stage workflow verbs (specify, constitution, clarify, plan, tasks, implement, validate, retrospect) are no longer exposed as user-facing CLI commands."* `cmdSpecify` is exported in `commands/specify.ts` but never imported by `cli.ts`. The slash command `/steel-specify` orchestrates the flow itself using helper subcommands (`steel state get`, `steel new-spec-id`, `steel branch-init`, `steel commit-step`, `steel render-prompt`, `steel run-gauge`, …); there is no top-level `steel specify` verb that scripts can invoke. As written, FR-8 is infeasible. **Suggested fix:** either (a) drop FR-8 entirely and confine the change to the slash-command surface plus a new helper subcommand the slash command can call (e.g., `steel state reset --from-completed --mode preserve|clean`), and update AC-7..AC-9 accordingly; or (b) explicitly include "register `steel specify` (and `steel clean --yes`) as user-facing CLI verbs" as in-scope work, with an FR that calls out the cli.ts wiring change. Pick one — leaving it ambiguous is the blocker.

- [B-2] FR-10 mandates a git commit (`steel(specify): reset state from completed workflow ...`) for the "y" path, but `state.json` is **gitignored** (`commands/clean.test.ts:90` and `commands/clean.ts:91-92` confirm: *"stateFile and tasksFile are gitignored"*). A commit "touching only `.steel/state.json`" will be empty. Furthermore, `commands/clean.ts:93` only commits when `config.autoCommit` is true — FR-10 currently demands the commit unconditionally. **Suggested fix:** either (a) drop the dedicated commit and instead require the audit trail be carried by the next forge commit (which is what FR-10 already half-says — pick one), or (b) explicitly state that the commit is best-effort and gated on `config.autoCommit`, and acknowledge that since `state.json` is gitignored the commit will only have content if a `previousSpecId` field or similar is added to a *non-ignored* artifact. The current text is impossible to implement as stated.

- [B-3] FR-3 is internally inconsistent and depends on a feature (resume-by-`--id`) that the spec itself flags as nonexistent in OQ-1 and inside FR-3's own NEEDS CLARIFICATION note. `src/spec-id.ts:21-48` confirms `--id` is a *new-spec prefix*, not a resume mechanism — it even *throws* if `specs/<customId>-<slug>/` already exists. So FR-3's "if `--id <X>` AND `specs/<X>-*` is in_progress, skip 0a" is describing behavior that cannot occur today (resume) AND would be blocked by an explicit collision check. The fallback "FR-3 reduces to: skip `0a` only when `--id` is provided" is also questionable — `--id` is a prefix for a *new* spec, so completed-state still applies; skipping `0a` whenever `--id` is present would silently let users start a new spec on top of completed state without being asked. **Suggested fix:** delete FR-3 in its current form. Decide one of: (i) `0a` always runs when state is completed regardless of `--id`, since `--id` is purely a prefix, OR (ii) add resume-by-`--id` as in-scope work with explicit FRs and ACs covering how it interacts with `generateSpecId`'s collision check. As written FR-3 is unimplementable and contradicts its own scope statement.

- [B-4] FR-1 #2 is logically a strict subset of FR-1 #1 in normal sequential operation, but the spec uses "either" framing without acknowledging that they collapse to one rule on well-formed state. Worse, the **trigger** chosen — "all seven stages complete OR currentStage==retrospect AND retrospect.status==complete" — does not match the user's verbatim description, which said: *"all stages complete OR currentStage == 'retrospect' with retrospect.status == 'complete'"*. The spec's choice arguably matches verbatim, but it omits a much simpler and more robust trigger consistent with how the existing precondition fails: **`currentStage != 'specification' || stages.specification.status != 'pending'` AND `stages.retrospect.status == 'complete'`**. As written, FR-1 misses the case where stages.retrospect.status is "complete" but currentStage has somehow been bumped past retrospect (impossible today, but the spec explicitly says it must be robust to "manually-edited state"). **Suggested fix:** Pick one canonical, minimal rule. I recommend: `state.stages.retrospect.status === 'complete'` — single condition, matches the user intent, robust to manual edits, no need for an OR clause.

### WARNING

- [W-1] **Overview misstates the current code.** The Overview says specify.ts *"requires `currentStage == 'specification'` AND `specification.status == 'pending'`"*. The actual check at `commands/specify.ts:28-35` is `if (currentStage !== 'specification' && specification.status !== 'pending') die(...)` — i.e., it dies only when *both* are not in the expected state. This is a logical OR-of-acceptance, not an AND-of-requirement. The behavioral conclusion the spec draws (completed state hard-fails) is correct, but the mechanical claim is wrong, and FR-8 says it will "replace the current hard-fail message in `commands/specify.ts:32-34`". A spec that's wrong about what it's replacing is a problem. **Suggested fix:** quote the actual condition or rewrite the description as "the current check rejects any state where currentStage is not 'specification' AND specification is not still 'pending'".

- [W-2] **Claude path's `AskUserQuestion` reference is alien to the existing canonical commands.** Searching `resources/commands/` shows zero existing uses of `AskUserQuestion`; the established pattern is *"ask the user verbatim"* followed by a quoted prompt (see `steel-specify.md:26-29`, `:96-99`, `:103`). FR-4's *"The Claude Code path SHOULD use `AskUserQuestion`"* introduces a new convention without justification or constitution alignment, and it conflicts with the parity requirement (NFR-3) — Codex and Gemini won't have `AskUserQuestion`. **Suggested fix:** drop the `AskUserQuestion` reference. Use the same "ask verbatim" idiom the rest of the file uses, and let provider-specific UI rendering be the providers' problem.

- [W-3] **AC-5 demands behavior `/steel-clean` doesn't deliver.** AC-5 says "on confirm produces the clean commit, removes prior tags and artifacts". But `commands/clean.ts:93` only commits *if `config.autoCommit` is true*, and the commit message format `commitStep('steel', 'clean', 1, ...)` produces `steel(clean): ...`, not the message AC-5 implies. AC-5 is also redundant with `commands/clean.test.ts` coverage and arguably out of scope (FR-6 says clean's behavior is unchanged; testing it again here belongs to clean's own tests). **Suggested fix:** narrow AC-5 to "Selecting **clean** invokes `/steel-clean`'s flow including its inner confirmation, and on completion proceeds to step 1." Drop the assertions about commit/tag/artifact details — those are clean's contract, not specify's.

- [W-4] **FR-9 conflates two different installed-copy paths.** The spec says the canonical update *"reflect the canonical change after a re-run of `steel-init` (or whatever sync mechanism `command-installer.ts` provides)"*. The constitution principle 3 plus the development guideline say canonical-source-first. But FR-9 hand-waves the propagation mechanism. The clarify stage will have to nail this down: does the spec require running `steel update` (which exists as a CLI command, line 71-73 of cli.ts), `steel-init`, or is the regeneration automatic? The vague wording will produce vague tests. **Suggested fix:** state explicitly which CLI verb propagates the change (almost certainly `steel update`) and have an AC that asserts the installed copy under `.claude/commands/steel-specify.md` matches the canonical source after running it.

- [W-5] **NFR-5's test list is incomplete.** It covers FR-1, FR-3, FR-5, FR-6, FR-7, FR-8 — but not FR-4 (prompt content), FR-9 (canonical-source propagation), or FR-10 (audit trail / commit behavior). Per the constitution coding standard, *"any feature that changes Forge/Gauge interaction should add or update tests that prove the observable behavior"*. **Suggested fix:** add explicit test coverage items for FR-4 (the slash-command prompt is the entry-point UX), FR-9 (a test verifying the installed copy is regenerable from canonical), and FR-10 (audit-trail commit, if it survives B-2's resolution).

- [W-6] **OQ-3 should not be open.** "Should `0a` fire when `state.json` is missing?" is answerable from existing code: `loadState` in `src/workflow.ts:98-117` recovers from committed artifacts when state.json is absent. If recovery yields a completed state (retrospect.status == complete, evidenced by tags or files), then yes, `0a` should fire — the *recovered* state will satisfy FR-1. The spec should decide this rather than punt. **Suggested fix:** state explicitly that FR-1 applies to the post-`loadState` state regardless of whether state.json was loaded fresh or recovered.

### NOTE

- [N-1] FR-5's reset-shape spec lists `currentStage`, `iteration`, per-stage fields, `specId`, `branch`, `baseBranch`, `description`, `skillsUsed` — that matches the `WorkflowState` in `src/workflow.ts:42-51` exactly except it does not say what to do with the optional `stages.X.iteration`/`startedAt`/`completedAt` fields when they are absent in the fresh shape. `createInitialState()` (workflow.ts:86-96) produces stages with only `{ status: 'pending' }` and no other keys. The spec says "clear" — be explicit: do you want them deleted (matching `createInitialState`'s output) or set to `undefined`? JSON-serialization-wise `delete` is correct.

- [N-2] FR-6 says `/steel-clean`'s inner confirmation must NOT be bypassed. Good. But FR-8's `--clean-completed` flag implies a non-interactive form. If `steel clean --yes` doesn't exist (it doesn't — see clean.ts), the non-interactive CLI parity demanded by NFR-3 cannot be fully satisfied. This is already flagged in OQ-2; if B-1 is resolved by adding the CLI, this becomes scope. If B-1 is resolved by removing the CLI, this becomes irrelevant.

- [N-3] FR-4 says `Other` is "not a valid response and MUST be rejected with the same prompt". This is a UX detail that's genuinely a clarify-stage call — and it's framed in `AskUserQuestion` terms (W-2). For an "ask verbatim" prompt the LLM has to enforce this in conversation handling. Worth confirming whether re-asking on invalid input is automatic or needs explicit prompt text.

- [N-4] OQ-4 (add `previousSpecId` field) is a reasonable design call the spec could have made itself. With B-2's commit-content problem, adding `previousSpecId` to a non-gitignored artifact (e.g., a row in `specs/<new>/spec.md`'s frontmatter or a dedicated `specs/<previous>/.steel-handoff.json`) is one path to making the audit trail real. Worth deciding rather than punting.

- [N-5] FR-7's exit message `"Cancelled. Run /steel-clean or /steel-specify when ready."` would be more useful if it included the previous specId, e.g., `"Cancelled. Previous workflow <X> is still recorded as complete. Run /steel-clean or /steel-specify when ready."`. Minor.

## Strengths

- The user-story-to-FR-to-AC mapping is tight and traceable; each US has at least one corresponding FR and at least one AC. Out-of-Scope section is unusually disciplined and prevents scope creep.
- The spec correctly identifies its own uncertainties as NEEDS CLARIFICATION inline rather than just punting in OQ; FR-3 and FR-8 have clarifying notes embedded where the reader needs them.
- The non-functional requirements actually reflect constitution principles (parity NFR-3, OS support NFR-4, atomicity NFR-2) instead of being a rote checklist.

## Verdict Reasoning

Four blocking issues (B-1 invents a CLI surface that doesn't exist; B-2 mandates a commit on a gitignored file; B-3 is internally contradictory; B-4 picks a more brittle FR-1 rule than necessary) make this spec unsafe to advance to clarification — the clarification stage would inherit factually wrong premises and produce work that can't ship as written. The warnings are individually fixable but cumulatively reinforce that the spec needs a pass against the actual code.

VERDICT: REVISE
