# Specification: Prompt User Before Reusing a Completed Workflow State in /steel-specify

**Spec ID:** 007-specify-prompt-on-completed-state
**Status:** Draft
**Created:** 2026-05-03
**Iteration:** 3 (clarifications applied: 3 OQs resolved, 2 warnings + 5 notes folded in)

## Overview

When a Steel-Kit project finishes a full workflow (ending with `retrospect.status == "complete"`), the next invocation of `/steel-specify` currently hard-fails. The check at `commands/specify.ts:28-35` —

```ts
if (state.currentStage !== 'specification' && state.stages.specification.status !== 'pending') die(...)
```

— rejects any state where `currentStage` has advanced past `specification` *and* the specification stage is no longer `pending`. After a completed retrospect, both conditions are true and the user must run `/steel-clean` to recover. That is non-obvious and discards prior artifacts/tags they may want to keep as audit history.

This spec adds an interactive precondition step **0a** to the canonical `/steel-specify` slash command (`resources/commands/steel-specify.md`) that detects a completed workflow in `.steel/state.json` and asks the user how to proceed:

- **y** — fresh state, preserve prior `specs/<previous-specId>/` and `steel/<previous-specId>/*-complete` tags as audit history.
- **clean** — invoke `/steel-clean` first (destructive reset).
- **cancel** — stop.

This is a slash-command-surface change. Per `src/cli.ts:48-52`, per-stage workflow verbs (specify, clarify, plan, …) are **not** exposed as user-facing CLI commands — the slash command is the only end-user entry point, and CI uses `steel run-all`. Constitution principle 3 (provider parity across Codex / Gemini CLI / Claude Code) is therefore satisfied by editing the canonical slash-command source; per-provider installed copies are regenerated by `steel update`.

The change does not alter Forge-Gauge iteration logic, stage advancement, artifact paths, or git-tag semantics.

## User Stories

- **US-1:** As a developer who just finished one feature workflow, I want `/steel-specify "next feature"` to ask me how to continue rather than failing with a stage-mismatch error, so that I do not have to memorize the recovery procedure.
- **US-2:** As a developer who wants to keep the prior spec's directory and `steel/*/*-complete` tags as audit history, I want a "fresh start, preserve history" option, so that starting feature 008 does not erase feature 007.
- **US-3:** As a developer who wants a true clean slate (no leftover artifacts, no prior tags), I want to invoke `/steel-clean` from inside the prompt without re-running another command, so that the workflow restart is one step.
- **US-4:** As a reviewer auditing the workflow later, I want to see in committed artifacts which prior specId a given new spec was started after, so that the chain of work is reconstructable from git history alone (constitution principle 4).

> Provider-parity coverage (constitution principle 3) is intentionally a non-functional concern (NFR-3), not a separate user story; all three providers consume the same canonical slash-command source.

## Functional Requirements

### FR-1: Detect a "previously-completed workflow"

After `loadState` (`src/workflow.ts:98-117`) returns the in-memory `WorkflowState`, classify the workflow as **completed** when:

```
state.stages.retrospect.status === 'complete'
```

This single condition subsumes the user's verbatim "all stages complete OR currentStage==retrospect with retrospect.status==complete" framing — `retrospect` is the last stage in `STAGE_ORDER` (`src/workflow.ts:53-61`), so a `complete` retrospect implies all earlier stages are complete in any well-formed state. The single-condition rule is also robust to manually-edited state where `currentStage` may have been advanced past `retrospect`.

The classification operates on the loaded state regardless of whether `state.json` was read fresh from disk or recovered by `loadState` from committed artifacts and git metadata — both paths produce the same `WorkflowState` shape (constitution constraint: state recovery must work from committed artifacts).

### FR-2: Trigger order — step 0a sits between prerequisite check and step 1

The new step `0a` runs:

- **After** the existing prerequisite block in `resources/commands/steel-specify.md` (verify `.steel/` exists, constitution is real, `state.json` loadable).
- **Before** step 1 (read state + config) of that file.

If FR-1 returns **not-completed** (`retrospect.status !== 'complete'`), step `0a` is a no-op and the existing flow proceeds unchanged. The check adds zero perceptible latency on the common path because it consults the already-loaded `WorkflowState` and does no additional I/O.

### FR-3: Interactive prompt content

When `0a` triggers, the slash command MUST present this exact prompt to the user, verbatim, with `<previous specId>` substituted from `state.specId`:

> A previous workflow (`<previous specId>`) is fully complete. Start a new workflow with this prompt? [y / clean / cancel]

The slash command uses the same "ask the user verbatim" idiom as the rest of `resources/commands/steel-specify.md` (e.g., lines 26-29, 96-99, 103). It does NOT use any provider-specific UI primitive (e.g., `AskUserQuestion`); rendering is the provider runtime's responsibility.

The three options have the following meanings:

- **y** — preserve-history reset (FR-4).
- **clean** — invoke `/steel-clean` first (FR-5).
- **cancel** — stop (FR-6).

Any response other than exactly `y`, `clean`, or `cancel` (case-insensitive, leading/trailing whitespace ignored) MUST cause the slash command to re-display the prompt verbatim. The slash command does not silently accept or interpret near-matches.

### FR-4: "y" — preserve-history reset

When the user selects **y**:

1. Capture `previousSpecId = state.specId` (string; may be `undefined` only in pathologically corrupt state, in which case skip step 4 below).
2. Replace the in-memory `state` with the output of `createInitialState()` (`src/workflow.ts:86-96`) — this produces a `WorkflowState` with `currentStage: 'specification'`, `iteration: 1`, every stage as `{ status: 'pending' }` (no `startedAt`, `completedAt`, or per-stage `iteration` keys), and no `specId`, `branch`, `baseBranch`, `description`, or `skillsUsed` fields.
3. Persist via `saveState(projectRoot, state)`.
4. **Do NOT** delete or modify `specs/<previousSpecId>/` directory, `.steel/tasks.json`, or any git tags (including `steel/<previousSpecId>/*-complete`).
5. Proceed to step 1 of `/steel-specify` (which generates a new spec ID, branch, and spec directory for the user's new description).

The audit trail for this reset is carried by FR-7, not by a dedicated commit.

### FR-5: "clean" — invoke /steel-clean first

When the user selects **clean**:

1. Invoke `/steel-clean` end-to-end, including its own confirmation prompt (`resources/commands/steel-clean.md` step 4: "This will remove iteration artifacts for spec `<specId>` and reset workflow state. Continue?"). The `/steel-specify` flow MUST NOT bypass that confirmation.
2. **Snapshot before invoking:** capture `previousSpecId = state.specId` (a non-empty string, since FR-1's trigger requires a completed workflow with `specId` set).
3. **Detect outcome by state diff:** after `/steel-clean` returns, re-read `.steel/state.json`. The outcome classification is:
   - **Successful clean** — reloaded state has `specId === undefined` (or equivalently `currentStage === "specification"` AND `stages.specification.status === "pending"`). Proceed to step 5. Note: this classification holds even if `cmdClean`'s optional `autoCommit` block (`commands/clean.ts:93-101`) failed *after* writing fresh state — the user-visible outcome (fresh state, artifacts removed) matches success, so `/steel-specify` proceeds. A failed autoCommit is a separate concern surfaced by `cmdClean` itself.
   - **Declined or failed-before-state-write** — reloaded state still has `specId === previousSpecId`. Treat as "did not complete" and go to step 4.
4. **Abort path:** print `/steel-clean did not complete — re-run /steel-specify when ready.` and exit. Do NOT silently fall back to "y" or any other option. No branch is created. No commits are added.
5. **Continue path:** proceed to step 1 of `/steel-specify` (which generates a new spec ID and branch from the user's description).

### FR-6: "cancel" — stop

When the user selects **cancel**:

1. Make no changes to `.steel/state.json`, no commits, no branch creation, no tag operations.
2. Print: `Cancelled. Previous workflow <previous specId> is still recorded as complete. Run /steel-clean or /steel-specify when ready.`
3. Exit.

### FR-7: Audit trail — `previousSpecId` in the new spec's frontmatter

When the **y** path proceeds to step 1 and a new spec ID is generated, the new spec directory's `spec.md` MUST include a `Previous Spec ID:` field in its header block:

```markdown
# Specification: <title>

**Spec ID:** <new-spec-id>
**Previous Spec ID:** <previousSpecId>
**Status:** Draft
**Created:** <YYYY-MM-DD>
```

**Placement:** insert immediately after `**Spec ID:**` and before `**Status:**`. The field is appended only when the **y** path of step `0a` was taken; specs created without a completed-workflow reset omit the line entirely. No `templates/spec.md` change is required because that template does not contain the header fields — they are Forge-LLM-added per spec.

**Rationale:** `.steel/state.json` is gitignored (`commands/clean.test.ts:90`, `commands/clean.ts:91-92`), so a commit touching only `state.json` would be empty. Committing `previousSpecId` into the new spec's tracked `spec.md` yields a real, reviewable audit record per constitution principle 4.

**Commit behavior:**
- If `config.autoCommit === true`, the existing forge commit at the end of step 6 (forge phase, iteration 1) carries the `Previous Spec ID:` line naturally — no additional commit is required.
- If `config.autoCommit === false`, the field is still written to `spec.md` (so it appears in `git status` and is captured by any subsequent commit), but `/steel-specify` does NOT add an automatic commit. This honors the user's `autoCommit: false` configuration per constitution principle 6 (automation is subordinate to user control).

### FR-8: Canonical-source update and per-provider propagation

The change MUST be applied to the canonical command source `resources/commands/steel-specify.md` only. Installed copies (`.claude/commands/steel-specify.md`, `.agents/skills/steel-specify/SKILL.md`, and any Gemini/Codex equivalents written by `command-installer.ts`) MUST NOT be edited directly.

Propagation is via `steel update` (`src/cli.ts:69-72`, command `cmdUpdate`), which is the existing CLI verb that refreshes Claude/Gemini/Codex command files. Users running an updated Steel-Kit will pick up step `0a` after running `steel update` in their project.

### FR-9: No CLI surface added

This spec does NOT add a `steel specify` user-facing CLI command, nor `--reset-completed` / `--clean-completed` flags. Per `src/cli.ts:48-52`, per-stage verbs are not exposed at the CLI; `cmdSpecify` exists only as an internal helper. Headless / CI users continue to use `steel run-all`. If a future spec wants a fully non-interactive entry point for completed-state recovery, that is its scope, not this one's.

### FR-10: Internal TypeScript helper for the classification rule

Add `isCompletedWorkflow(state: WorkflowState): boolean` as an internal TypeScript helper in `src/workflow.ts`, implementing the FR-1 rule. The helper exists for TS-side callers and tests, ensuring the rule can be asserted directly without shelling out.

The slash command implements the FR-1 rule inline via the existing `steel state get --field stages.retrospect.status` primitive (literal compare to `complete`); no new CLI subcommand is added. **Both call sites — the bash-level slash command and the TS helper — must change together if FR-1 ever evolves.** This is an accepted duplication in service of minimal scope (no new CLI surface) and matches Steel-Kit's existing pattern of slash commands consulting state via existing helper subcommands.

## Non-Functional Requirements

- **NFR-1:** FR-1 classification MUST be a pure function of `WorkflowState`; no I/O beyond the existing `loadState` call.
- **NFR-2:** The "y" reset (FR-4) MUST be atomic from the user's perspective: either the new state is fully written to `.steel/state.json` or the file is unchanged. Partial writes (e.g., currentStage reset but stages still containing completed timestamps) are not acceptable. `saveState` already provides this; the FR is a constraint on how callers use it.
- **NFR-3:** Behavior parity across Codex, Gemini CLI, and Claude Code provider paths is required. Because the change is exclusively in the canonical slash-command source, parity is achieved by `steel update` regenerating per-provider copies. No provider-specific branches in the command logic.
- **NFR-4:** Supported OSes remain macOS and Linux only; no Windows-specific assumptions in any new prompt or shell-out.
- **NFR-5:** Tests MUST cover, at minimum:
  - **FR-1 / FR-10:** truth table for `isCompletedWorkflow` — `retrospect.status` of `complete` returns true; `pending` / `in_progress` return false; absence of `state.stages.retrospect` (corrupt input) returns false (no throw).
  - **FR-3:** prompt text matches the verbatim string in FR-3.
  - **FR-4:** post-reset `state.json` is byte-identical to `createInitialState()` output (snapshot test); `specs/<previousSpecId>/` exists unchanged before vs after; `git tag --list 'steel/<previousSpecId>/*'` returns the same set before vs after.
  - **FR-5:** when `/steel-clean`'s inner confirmation is declined, the FR-5 step 2.5 state-diff check correctly identifies the decline and `/steel-specify` aborts with the step-4 message; no branch is created.
  - **FR-6:** "cancel" yields a byte-identical `state.json` and zero new commits.
  - **FR-7:** when the **y** path was taken, the new `spec.md` contains a `**Previous Spec ID:** <X>` line placed between `**Spec ID:**` and `**Status:**`; when the **y** path was NOT taken, the line is absent.
  - **FR-8:** running `steel update` in a project after a Steel-Kit upgrade copies the updated `resources/commands/steel-specify.md` to the per-provider installed paths (this can be a thin assertion against `command-installer.ts` since the propagation path is existing functionality).

## Acceptance Criteria

- **AC-1:** With `state.stages.retrospect.status == "complete"`, running `/steel-specify "feature X"` displays the FR-3 prompt verbatim and does NOT proceed until a valid choice is made.
- **AC-2:** With `state.stages.retrospect.status == "in_progress"` or `"pending"`, the prompt does NOT appear and the existing flow runs unchanged.
- **AC-3:** With state classified as completed and the user input is whitespace, an unrecognized token, or empty, the prompt is re-displayed verbatim until a valid choice is given.
- **AC-4:** Selecting **y** results in `state.json` byte-identical to `createInitialState()` output; `specs/<previousSpecId>/` is intact; `git tag --list 'steel/<previousSpecId>/*'` returns the prior tags unchanged; the new branch is created; the new `spec.md` contains a `**Previous Spec ID:** <previousSpecId>` line.
- **AC-5:** Selecting **clean** invokes `/steel-clean`'s flow (including its inner confirmation) and on completion proceeds to step 1.
- **AC-6:** Selecting **clean** then declining the inner `/steel-clean` confirmation aborts `/steel-specify` with the FR-5 step-4 message; no branch is created; no commits are added.
- **AC-7:** Selecting **cancel** leaves `.steel/state.json` byte-identical, creates no commits, no branch, no tag operations, and prints the FR-6 message.
- **AC-8:** Running `steel update` in a project after pulling a Steel-Kit version that contains step `0a` in `resources/commands/steel-specify.md` copies the updated content into `.claude/commands/steel-specify.md` (and the equivalent Gemini/Codex paths if those provider toolchains are installed).
- **AC-9:** No existing test in `commands/clean.test.ts`, `src/workflow.test.ts`, `src/git-config.test.ts`, or `src/spec-id.test.ts` fails unless the failure is a deliberate snapshot update reviewed against this spec.

## Out of Scope

- Auto-detecting completed workflows from git tags alone when `state.json` is malformed beyond `loadState`'s recovery — that is a `/steel-doctor` problem.
- Adding a "resume by `--id`" feature. `src/spec-id.ts:21-50` confirms `--id` is purely a new-spec prefix and explicitly throws when `specs/<id>-<slug>/` already exists; resume is not a concept the codebase supports today, and adding it is a separate spec.
- Adding `steel specify` (or any per-stage verb) as a user-facing CLI command. `src/cli.ts:48-52` documents the deliberate decision to exclude these; this spec respects that contract.
- Adding `steel clean --yes` for non-interactive cleaning. The "clean" path under FR-5 explicitly preserves the inner confirmation; if a non-interactive clean is wanted, that is a separate spec.
- Changing `/steel-clean`'s own behavior, commit message format, tag-removal scope, or interactive confirmation.
- Changing how `command-installer.ts` propagates canonical sources to per-provider copies.
- A "rename and archive" option that moves `specs/<previous>/` to `specs/archive/<previous>/`. Separate spec if desired.
- Changing the existing precondition error in `commands/specify.ts:28-35` (the CLI helper). It remains as a defensive check for misconfiguration; the new step `0a` lives in the slash-command flow upstream of it.
- Pre-emptive slug-collision detection in step `0a`. The existing throw in `src/spec-id.ts:46-50` produces the right error when `--id <X>` plus the new slug collides with an existing spec dir. Auto-numbered specs (no `--id`) cannot collide because `generateSpecId` picks `nextNum = max(existing) + 1`.
- Adding `Previous Spec ID:` to downstream artifacts (`tasks.md`, `plan.md`, `retrospect.md`). `spec.md` is the canonical audit anchor; downstream artifacts reach the prior spec via `git log` through the new spec dir.
