BLOCKING: FR-9, FR-11, FR-13, and AC-5 are internally inconsistent about what counts as a canonical input to generated surfaces. The spec says doctor must verify `resources/commands/`, `prompts/`, and `templates/`, and AC-5 says changing canonical `command/prompt/template` content should make generated surfaces stale. But the actual generator only reads `resources/commands/*.md` when producing Claude, Gemini, and Codex surfaces ([src/command-installer.ts](/Users/ezchi/Projects/steel-kit/src/command-installer.ts#L13), [src/command-installer.ts](/Users/ezchi/Projects/steel-kit/src/command-installer.ts#L61), [src/command-installer.ts](/Users/ezchi/Projects/steel-kit/src/command-installer.ts#L75), [src/command-installer.ts](/Users/ezchi/Projects/steel-kit/src/command-installer.ts#L95)). As written, a prompt/template edit has no defined rendered output to compare against, so the staleness check is not implementation-ready or testable.

BLOCKING: FR-7 claims to use “the same stage/file mapping as the current workflow runtime,” but the listed rules do not match the runtime. The runtime maps one file per stage for recovery, and there is no primary `implementation` stage file at all ([src/workflow.ts](/Users/ezchi/Projects/steel-kit/src/workflow.ts#L178), [src/workflow.ts](/Users/ezchi/Projects/steel-kit/src/workflow.ts#L458)). The spec instead defines cumulative prerequisites and treats `implementation` as requiring prior documents. That may be a reasonable diagnostic policy, but it is not “the same mapping,” so the source of truth is ambiguous. This directly affects completeness, consistency, and tests.

BLOCKING: The severity model is underspecified for several core checks. FR-18 through FR-22 require deterministic `pass`/`warn`/`fail` behavior and exit codes, but only provider CLI checks have explicit severity rules. The spec does not define severity for missing initialization files, placeholder constitution, branch/state/spec drift, missing stage files, missing generated surfaces, stale generated surfaces, missing canonical sources, or recoverable missing state. Without a severity matrix, `--json` output and CI behavior are not stable enough for automation, which conflicts with the constitution’s auditability and parity requirements.

WARNING: Drift detection is incomplete because FR-6 ignores `state.branch`, even though the workflow persists branch into state and exposes it as part of workflow status ([resources/commands/steel-specify.md](/Users/ezchi/Projects/steel-kit/resources/commands/steel-specify.md#L18), [src/workflow.ts](/Users/ezchi/Projects/steel-kit/src/workflow.ts#L42), [commands/status.ts](/Users/ezchi/Projects/steel-kit/commands/status.ts#L41)). A corrupted or stale `state.branch` would evade doctor despite the stated goal of detecting drift between state, branch, and spec context.

WARNING: FR-8 is too vague to be reliably testable. It says doctor should report when state files are absent but artifacts or branch context suggest recoverable state, yet it does not define the recovery algorithm, which artifacts/tags count, whether `.steel/tasks.json` matters, or how to classify partial recovery. The current recovery logic is heuristic and tied to a mutating loader path ([src/workflow.ts](/Users/ezchi/Projects/steel-kit/src/workflow.ts#L86), [src/workflow.ts](/Users/ezchi/Projects/steel-kit/src/workflow.ts#L103)), so a read-only doctor needs a more explicit rule set.

NOTE: The spec is generally well-scoped to Issue 1 and stays aligned with the constitution on read-only behavior, provider parity, and auditability. The main problem is not direction; it is that several rules are still too ambiguous or contradictory to be safely implemented as a shared diagnostic command.
